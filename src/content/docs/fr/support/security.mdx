OculiX tourne entièrement sur votre machine. Rien ne phone home, aucune télémétrie n'est envoyée, aucun analytics, aucun appel d'auto-update. Cette page documente la posture sécurité du projet pour les organisations qui doivent évaluer OculiX avant déploiement en environnement régulé.

<Aside type="tip" title="La version courte">
  Pas de SaaS à attaquer. Pas de dépendance cloud. MIT, source-available, audit trail signé disponible via le module MCP. On publie ce qu'on a, et on est explicite sur ce qu'on n'a pas.
</Aside>

## Ce qu'on publie aujourd'hui

<CardGrid>
  <Card title="Politique de divulgation de vulnérabilités" icon="warning">
    Pour tout ce qui est sensible — bypass d'auth, RCE, fuite de credentials, supply chain — **n'ouvrez pas d'issue publique**. Utilisez le [signalement privé de vulnérabilité de GitHub](https://github.com/oculix-org/Oculix/security/advisories/new).

    **Timeline de réponse :**
    - Accusé sous 48 heures
    - Triage et sévérité sous une semaine
    - Calendrier de fix communiqué en transparence
    - Divulgation coordonnée si in-the-wild

    Politique complète dans [SECURITY.md](https://github.com/oculix-org/Oculix/blob/master/SECURITY.md).
  </Card>

  <Card title="Licence MIT, pour toujours" icon="approve-check">
    Le cœur OculiX est sous licence MIT. **La licence est irrévocable.** Il n'existe pas de version d'OculiX où le cœur devient propriétaire. Chaque ligne expédiée est dans le repo public, auditable, forkable, signable par vous.

    ![MIT](https://img.shields.io/badge/Licence-MIT-green?style=flat-square)
  </Card>

  <Card title="Journal d'audit MCP" icon="document">
    Le module serveur MCP optionnel écrit un **journal d'audit JSONL signé Ed25519 et chaîné SHA-256** — conçu pour les environnements où chaque action automatisée doit être tamper-evident de bout en bout. Chaque entrée référence le hash de la précédente, donc la modification rétroactive d'un enregistrement invalide toutes les signatures suivantes.

    Le plus souvent cité par les utilisateurs en défense, banque, santé.

    ![Ed25519](https://img.shields.io/badge/Signature-Ed25519-blueviolet?style=flat-square)
    ![SHA-256](https://img.shields.io/badge/Cha%C3%AEne-SHA--256-blueviolet?style=flat-square)
  </Card>
</CardGrid>

## Choix d'architecture qui affectent la sécurité

Ce ne sont pas des fonctionnalités qu'on met en avant — ce sont les conséquences de la façon dont OculiX est construit.

<CardGrid stagger>
  <Card title="Pas de dépendance cloud" icon="laptop">
    L'IDE, la bibliothèque API, le runner de scripts marchent tous **complètement offline**. Le seul accès réseau est ce que votre script demande explicitement.
  </Card>

  <Card title="Pas de télémétrie" icon="approve-check">
    On ne collecte aucune donnée d'usage, aucun rapport d'erreur, aucune métrique de performance, rien. Si vous optez pour les vérifications de mise à jour, OculiX interroge GitHub Releases pour une nouvelle version — **c'est le seul appel sortant**, et il est opt-in.
  </Card>

  <Card title="Pas d'auto-update" icon="information">
    Les mises à jour arrivent quand vous téléchargez explicitement une nouvelle release. **Pas de patchs silencieux.** Ce qui est expédié, c'est ce qui tourne, jusqu'à ce que vous en décidiez autrement.
  </Card>

  <Card title="Source-available partout" icon="github">
    Chaque ligne qui tourne est dans le [repo public](https://github.com/oculix-org/Oculix). Vous pouvez l'auditer, la forker, la builder depuis les sources, signer vos propres builds.
  </Card>
</CardGrid>

## Environnements régulés

OculiX a été déployé par des organisations en secteurs régulés — **banque, défense, santé, gouvernement** — principalement parce qu'il tourne on-premises sans appel tiers. Le projet lui-même n'est pas certifié (SOC 2, HIPAA, FedRAMP), mais son architecture facilite son intégration dans un environnement certifié que vous contrôlez déjà.

<Aside type="caution" title="Honnête sur les certifications">

On n'est certifié pour aucune de ces normes. **Aucune page OculiX ne prétendra l'inverse.**

![SOC 2](https://img.shields.io/badge/SOC%202-Non%20certifi%C3%A9-lightgrey?style=flat-square)
![HIPAA](https://img.shields.io/badge/HIPAA-Non%20certifi%C3%A9-lightgrey?style=flat-square)
![FedRAMP](https://img.shields.io/badge/FedRAMP-Non%20certifi%C3%A9-lightgrey?style=flat-square)
![ISO 27001](https://img.shields.io/badge/ISO%2027001-Non%20certifi%C3%A9-lightgrey?style=flat-square)

L'architecture du projet — pas de télémétrie, pas de cloud, pas d'appels API tiers depuis le runtime — facilite son intégration dans un environnement certifié que **vous** opérez. La certification appartient à votre déploiement, pas à nous.

</Aside>

Arrangements courants qu'on a vus avec les utilisateurs régulés :

<CardGrid>
  <Card title="Déploiement air-gap" icon="laptop">
    Build depuis les sources sur votre propre infrastructure, mirror des artefacts en interne, pas besoin d'internet après le build initial. L'architecture supporte ça out of the box.
  </Card>

  <Card title="Builds reproductibles" icon="approve-check">
    Les artefacts Maven sur Central sont produits par le même workflow que les releases GitHub — vous pouvez vérifier en relançant le workflow sur votre propre infra.
  </Card>

  <Card title="Fenêtre de divulgation sur mesure" icon="document">
    Si vous trouvez un problème de sécurité et avez besoin d'une fenêtre de divulgation coordonnée **plus longue que les 90 jours par défaut** pour patcher en interne, on l'honore.
  </Card>

  <Card title="Walkthrough CISO" icon="comment-alt">
    Un walkthrough du chemin de code des modules pertinents avec votre équipe sécurité, planifié par email. Pas formalisé, mais arrivé plusieurs fois.
  </Card>
</CardGrid>

Pour des packs de documentation procurement-grade (lettres d'indemnisation, énoncés d'architecture sécurité écrits, walkthroughs du log d'audit), écrivez à [contact@oculix.org](mailto:contact@oculix.org).

## Ce que cette page n'est pas

<Aside type="note">
- **Pas une liste de certifications marketing.** On publie ce qu'on a et on est explicite sur ce qu'on n'a pas.
- **Pas un substitut à votre assessment interne.** On publie des artefacts pour faciliter cet assessment, pas pour le remplacer.
- **Pas une page sécurité SaaS.** Il n'y a pas de service à attaquer depuis l'extérieur ; le modèle de menace est « du code qui tourne sur votre machine dans votre environnement ».
</Aside>
