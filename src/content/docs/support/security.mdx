OculiX runs entirely on your machine. Nothing phones home, nothing sends telemetry, no analytics, no auto-update calls. This page documents the project's security posture for organizations that need to assess OculiX before deployment in regulated environments.

<Aside type="tip" title="The short version">
  No SaaS to attack. No cloud dependency. MIT, source-available, signed audit trail available via the MCP module. We publish what we have, and we're explicit about what we don't.
</Aside>

## What we publish today

<CardGrid>
  <Card title="Vulnerability disclosure policy" icon="warning">
    For anything sensitive — auth bypass, RCE, credential leakage, supply chain — **do not open a public issue**. Use GitHub's [private vulnerability reporting](https://github.com/oculix-org/Oculix/security/advisories/new).

    **Response timeline:**
    - Acknowledgement within 48 hours
    - Triage and severity within one week
    - Fix timeline communicated transparently
    - Coordinated disclosure when in-the-wild

    Full policy in [SECURITY.md](https://github.com/oculix-org/Oculix/blob/master/SECURITY.md).
  </Card>

  <Card title="MIT license, forever" icon="approve-check">
    The OculiX core is MIT-licensed. **The license is irrevocable.** There's no version of OculiX where the core becomes proprietary. Every line that ships is in the public repository, auditable, forkable, signable by you.

    ![MIT](https://img.shields.io/badge/License-MIT-green?style=flat-square)
  </Card>

  <Card title="MCP audit journal" icon="document">
    The optional MCP server module writes an **Ed25519-signed, SHA-256-chained JSONL audit journal** — designed for environments where every automated action needs to be tamper-evident end to end. Each entry references the hash of the previous one, so retroactive modification of any record invalidates all subsequent signatures.

    Most often cited by users in defense, banking, healthcare.

    ![Ed25519](https://img.shields.io/badge/Signature-Ed25519-blueviolet?style=flat-square)
    ![SHA-256](https://img.shields.io/badge/Chain-SHA--256-blueviolet?style=flat-square)
  </Card>
</CardGrid>

## Architecture choices that affect security

These aren't features we promote — they're consequences of how OculiX is built.

<CardGrid stagger>
  <Card title="No cloud dependency" icon="laptop">
    The IDE, the API library, the script runner all work **fully offline**. The only network access is what your script explicitly requests.
  </Card>

  <Card title="No telemetry" icon="approve-check">
    We don't collect usage data, error reports, performance metrics, or anything else. If you opt into update checks, OculiX queries GitHub Releases for a new version — **that's the only outbound call**, and it's opt-in.
  </Card>

  <Card title="No auto-update" icon="information">
    Updates happen when you explicitly download a new release. **No silent patches.** What ships is what runs, until you decide otherwise.
  </Card>

  <Card title="Source-available everything" icon="github">
    Every line that runs is in the [public repository](https://github.com/oculix-org/Oculix). You can audit it, fork it, build it from source, sign your own builds.
  </Card>
</CardGrid>

## Regulated environments

OculiX has been deployed by organizations in regulated sectors — **banking, defense, healthcare, government** — primarily because it runs on-premises with no third-party calls. The project itself isn't certified (SOC 2, HIPAA, FedRAMP), but its architecture makes it easier to fit into a certified environment you already control.

<Aside type="caution" title="Honest about certifications">

We are not certified for any of these. **No OculiX page will ever claim otherwise.**

![SOC 2](https://img.shields.io/badge/SOC%202-Not%20certified-lightgrey?style=flat-square)
![HIPAA](https://img.shields.io/badge/HIPAA-Not%20certified-lightgrey?style=flat-square)
![FedRAMP](https://img.shields.io/badge/FedRAMP-Not%20certified-lightgrey?style=flat-square)
![ISO 27001](https://img.shields.io/badge/ISO%2027001-Not%20certified-lightgrey?style=flat-square)

The project's architecture — no telemetry, no cloud, no third-party API calls from the runtime — makes it easier to fit into a certified environment **you** operate. The certification belongs to your deployment, not to us.

</Aside>

Common arrangements we've seen with regulated users:

<CardGrid>
  <Card title="Air-gapped deployment" icon="laptop">
    Build from source on your own infrastructure, mirror artefacts internally, no internet required after the initial build. The architecture supports this out of the box.
  </Card>

  <Card title="Reproducible builds" icon="approve-check">
    The Maven artefacts on Central are produced by the same workflow as the GitHub releases — you can verify by re-running the workflow on your own infrastructure.
  </Card>

  <Card title="Custom disclosure window" icon="document">
    If you find a security issue and need a coordinated disclosure window **longer than the default 90 days** for internal patching, we honor that.
  </Card>

  <Card title="CISO walkthrough" icon="comment-alt">
    A code-path walkthrough of the relevant modules with your security team, scheduled by email. Not formalized, but it has happened multiple times.
  </Card>
</CardGrid>

For procurement-grade documentation packs (indemnification letters, written security architecture statements, audit log walkthroughs), reach out at [contact@oculix.org](mailto:contact@oculix.org).

## What this page is not

<Aside type="note">
- **Not a marketing certification list.** We publish what we have and we're explicit about what we don't.
- **Not a substitute for your own internal assessment.** We publish artefacts to make that assessment easier, not replace it.
- **Not a SaaS security page.** There's no service to attack from outside; the threat model is "code that runs on your machine in your environment".
</Aside>
